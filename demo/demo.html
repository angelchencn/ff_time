## Formula Structure Convention

Every generated formula MUST follow the standard Oracle Fast Formula structure
used in production environments. This includes:

### 1. Header Comment Block (REQUIRED)

Always generate a professional header comment block at the top:

/******************************************************************************
*
* Formula Name : <FORMULA_NAME>
*
* Formula Type : <FORMULA_TYPE>
*
* Description : <Brief description of what the formula does>
*
* Change History
* --------------
*
* Who Ver Date Description
* --------------- ---- ----------- --------------------------------
* Payroll Admin 1.0 <YYYY/MM/DD> Created.
*
******************************************************************************/

### 2. Formula Naming Convention

Formula names follow the pattern: <PREFIX>_<BUSINESS_DESCRIPTION>_<SUFFIX>

Common suffixes: _EARN, _RESULTS, _PRORATION, _BASE,
_CALCULATOR, _DISTRIBUTION, _DEDN, _AUTO_INDIRECT

Common prefixes: ORA_WFM_, ORA_HRX_, WFM_, HRX_
Time rule subtypes: _TCR_, _TER_, _TSR_, _TDR_

### 3. Formula Body Order (REQUIRED)

Formulas MUST follow this strict ordering:
1. Header comment block
2. DEFAULT statements (with type annotations where needed)
3. INPUTS ARE statement
4. Variable assignments and business logic
5. Logging calls (PAY_INTERNAL_LOG_WRITE) at entry/exit
6. RETURN statement

### 4. End Marker

Optionally end the formula with: /* End Formula Text */

## Output Format

- Return only valid Fast Formula code unless explicitly asked for an explanation.
- Use consistent 2-space indentation for nested blocks.
- Include DEFAULT statements for all input variables.
- End every formula with a RETURN statement.
- ALWAYS include the header comment block as described above.

## DEFAULT Value Type Rules

Choose the correct DEFAULT value based on the variable name:
- Use string defaults (' ') for: NAME, TEXT, DESC, CODE, TYPE, STATUS, FLAG,
  MESSAGE, MSG, LABEL, CATEGORY, TITLE, MODE, REASON, COMMENT, NOTE, KEY...
- Use numeric defaults (0) for: COUNT, TOTAL, AMOUNT, RATE, HOURS, DAYS,
  SALARY, PAY, WAGE, BALANCE, QTY, NUMBER, PCT, PERCENT, FACTOR, LIMIT...
- Use date defaults ('01-JAN-0001'(DATE)) for: DATE, START, END, EFFECTIVE,
  EXPIRY, HIRE, TERMINATION, BIRTH...

Context Window Impact on Output Quality

When input approaches the model's context limit, output quality degrades — even if the response is not truncated.
Effect Impact
Attention Dilution Transformer attention spreads across all input tokens. At 100K tokens each token gets less weight — the model is more likely to “forget” earlier instructions (e.g., CRITICAL REQUIREMENTS at token 3K lose influence by token 100K).
Lost in the Middle Research confirms LLMs recall the beginning and end of context best; middle sections are most likely ignored. A 65K DBI reference list placed mid-context may be largely skipped.
Reasoning Degradation Larger KV-cache reduces per-step computation precision (especially on quantized models like FP8). Manifests as: logic errors, syntax rule violations (ELSIF→ELSEIF regressions), wrong RETURN contracts.
Output Compression Less room for output forces the model to “rush” — skipping header comments, omitting PAY_INTERNAL_LOG_WRITE, shortening RETURN variable lists.

Quality vs. Input Ratio (128K model)

Input Ratio Quality Typical Behavior
< 25% (~32K) Normal All rules followed correctly
25–50% (~32–64K) Slight decline Occasionally misses one rule (e.g., omits exit log)
50–75% (~64–96K) Noticeable decline Drops 2–3 rules; mid-context DBI entries ignored

Key Design Decisions

🧠 ADDITIONAL_PROMPT_TEXT — per-template prompt overlay
A CLOB column that the AI service appends to the system prompt whenever the template is selected. Lets domain-specific rule sets (e.g. Time & Labor override rules) ride with the template instead of living in code. Templates without extra guidance simply leave this NULL.
🔍 SEMANTIC_FLAG — RAG participation toggle
Y means this template is embedded into the vector store (ChromaDB today, Oracle 23ai Vector Search later) so it can be retrieved by semantic similarity. N excludes the row from retrieval while keeping it browseable by direct lookup — useful for drafts, internal templates, or near-duplicates.
🌱 Seed Data Framework integration
Full Fusion SDF participation via SEED_DATA_SOURCE, MODULE_ID, ORA_SEED_SET1/2, and ENTERPRISE_ID. Oracle-delivered templates flow through the standard seed extract/apply pipeline; upgrades refresh seeded rows without touching user-created ones.
🗝️ SOURCE_TYPE — seeded vs user-created discriminator
SEEDED rows are delivered by Oracle and read-only for end users (REST API rejects updates). USER_CREATED rows are fully mutable and saved via the POST /api/templates endpoint. Both live in the same table for unified retrieval.
